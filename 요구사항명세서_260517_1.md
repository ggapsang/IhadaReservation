# 요구사항 명세서

**프로젝트**: 예약 관리 시스템 - 결제 모듈 통합 선작업  
**대상 저장소**: IhadaReservation (현재 깃허브)  
**문서 버전**: 1.0  
**작성일**: 2026-05-17  
**상태**: 가계약 단계 (본 계약 진입 시 즉시 결제 모듈 통합 가능하도록 사전 정비)

---

## 1. 문서 개요

### 1.1 목적

본 문서는 현재 운영 중인 예약 관리 시스템에 토스페이먼츠 결제 모듈을 통합하기 위한 선작업 요구사항을 정의한다. 본 계약 체결 시점에 결제 모듈을 신속하게 결합할 수 있도록, 코드 구조와 데이터 스키마를 사전 정비한다.

### 1.2 명세 방식

본 문서는 **블랙박스 명세** 방식을 따른다.
- 외부 인터페이스(입력, 출력, 사이드 이펙트, 예외)를 정의한다
- 내부 구현 방식(알고리즘, 라이브러리 선택, 함수 분해 등)은 개발자 재량
- 단, 비기능 요구사항(보안, 성능, 확장성)은 반드시 충족

### 1.3 작업 범위 구분

| 단계 | 시점 | 범위 |
|---|---|---|
| Phase 0 (본 문서) | 가계약 ~ 본 계약 전 | 코드 정비, 스키마 확장, 인터페이스 정의 |
| Phase 1 | 본 계약 후 | 결제 모듈 구현, 옵션 상품 기능 |
| Phase 2 | 안정화 후 | 정기결제, 통계, 추가 기능 |

본 문서는 **Phase 0**만 다룬다.

### 1.4 용어 정의

| 용어 | 정의 |
|---|---|
| 예약 (Reservation) | 고객이 신청한 공간 대여 건 |
| 주문 (Order) | 결제를 위해 발급된 결제 단위 (예약 + 옵션 상품 포함) |
| 임시 점유 (Temp Hold) | 결제 진행 중인 시간대를 일시적으로 점유한 상태 |
| 결제 검증 (Verification) | 클라이언트 결제 완료 신호를 PG사 API로 재확인하는 과정 |
| PG (Payment Gateway) | 결제 대행사 (본 프로젝트는 토스페이먼츠) |
| Webhook | PG사가 우리 서버로 결제 결과를 비동기 통보하는 방식 |

## 2. 현황 진단

### 2.1 현재 시스템 구조

```
Code.gs (단일 백엔드 파일, ~700줄)
  ├─ WebApp 진입점 (doGet, doPost)
  ├─ 설정 관리 (getSettings, getRoomInfo)
  ├─ 가격 계산 (calculatePrice)
  ├─ 예약 가능 확인 (checkAvailability)
  ├─ 예약 등록 (submitReservation)
  ├─ Calendar 연동 (createCalendarEvent, syncToCalendar)
  ├─ 파일 업로드 (uploadFile)
  ├─ 약관 (getTermsContent)
  └─ 유틸리티

index.html (UI + 클라이언트 JS, ~1000줄)
```

### 2.2 통합 시 예상 충돌 지점

| 항목 | 현황 | 충돌 위험 |
|---|---|---|
| 단일 파일 비대화 | Code.gs 700줄 | 결제 모듈 추가 시 1500줄 이상 예상 |
| 예약 = 시트 1행 | 결제 정보 수용 불가 | 컬럼 추가 또는 시트 분리 필요 |
| 옵션 상품 개념 없음 | 가격은 룸+인원+시간만 고려 | 가격 계산 로직 전면 재설계 필요 |
| 입금 확인 = 수동 체크박스 | 자동 결제와 충돌 | 결제 상태 관리 체계 신설 필요 |
| 시크릿 키 관리 정책 부재 | 토큰/API키 추가 시 무질서 | Properties 표준화 필요 |
| 에러 처리 산발적 | 함수마다 다른 패턴 | 통일된 에러 응답 포맷 필요 |

### Task 0-3. 결제 모듈 인터페이스 정의 (Stub)

**목적**  
Phase 1에서 채워질 결제 함수의 시그니처를 미리 정의하여, 다른 영역(예약, UI)이 이를 호출하는 코드를 미리 작성 가능하게 함

**요구 결과**  
다음 함수들의 **빈 구현(stub)**을 작성한다. 시그니처와 반환 타입은 본 명세대로, 내부 로직은 임시 응답 또는 `NOT_IMPLEMENTED` 에러 반환으로 처리.

#### 3-1. `createPaymentOrder(formData)`

**입력**
| 파라미터 | 타입 | 필수 | 설명 |
|---|---|---|---|
| formData | Object | Y | 예약 폼 데이터 (기존 submitReservation 입력과 동일) |
| formData.options | Array<{id, quantity}> | N | 선택된 옵션 상품 |

**출력 (성공)**
```
{
  success: true,
  orderId: "uuid-string",
  amount: Number,
  amountBreakdown: {
    reservation: Number,
    options: Number,
    vat: Number,
    total: Number
  },
  expiresAt: ISOString
}
```

**출력 (실패)**
```
{
  success: false,
  errorCode: String,
  errorMessage: String
}
```

**사이드 이펙트**
- `임시주문` 시트에 새 행 추가 (상태: HOLDING)
- `결제로그` 시트에 ORDER_CREATED 이벤트 기록

**예외 케이스**
| 케이스 | errorCode |
|---|---|
| 필수 입력 누락 | INVALID_INPUT |
| 시간대 이미 점유됨 | TIME_BLOCKED |
| 옵션 상품 ID 미존재 | INVALID_OPTION |
| 시트 접근 실패 | STORAGE_ERROR |

**보안 요구사항**
- 금액은 반드시 서버에서 재계산 (클라이언트 전달값 사용 금지)
- orderId는 UUID v4 형식

#### 3-2. `verifyPayment(orderId, paymentKey)`

**입력**
| 파라미터 | 타입 | 필수 | 설명 |
|---|---|---|---|
| orderId | String | Y | createPaymentOrder가 발급한 주문번호 |
| paymentKey | String | Y | PG사가 발급한 결제 식별자 |

**출력 (성공)**
```
{
  success: true,
  reservationNumber: "RES20260517-001",
  amount: Number,
  paymentMethod: String
}
```

**출력 (실패)**
```
{
  success: false,
  errorCode: String,
  errorMessage: String
}
```

**사이드 이펙트**
- `예약내역` 시트에 새 행 추가 (결제 정보 포함)
- `임시주문` 시트의 해당 주문 상태를 COMPLETED로 변경
- `결제로그`에 PAYMENT_VERIFIED 기록
- Google Calendar 이벤트 생성
- 고객에게 예약 확정 이메일/알림톡 발송
- 관리자에게 새 예약 알림 발송

**예외 케이스**
| 케이스 | errorCode | 후처리 |
|---|---|---|
| 주문번호 미존재 | ORDER_NOT_FOUND | - |
| 주문 만료 | ORDER_EXPIRED | 결제 자동 취소 시도 |
| PG 금액 불일치 | AMOUNT_MISMATCH | 결제 자동 취소 시도 |
| PG API 호출 실패 | PG_API_ERROR | 재시도 (설정값에 따름) |
| PG 결제 미완료 | PAYMENT_NOT_COMPLETED | - |

**보안 요구사항**
- PG사 API 응답의 금액과 `임시주문` 저장 금액 일치 여부 필수 검증
- 금액 불일치 시 즉시 결제 취소 API 호출
- Webhook 시그니처 검증 (PG사 제공 secret 사용)

#### 3-4. `refundPayment(reservationNumber, amount, reason)`

**입력**
| 파라미터 | 타입 | 필수 | 설명 |
|---|---|---|---|
| reservationNumber | String | Y | 예약번호 |
| amount | Number | N | 환불 금액 (생략 시 전액) |
| reason | String | Y | 환불 사유 |

**출력**
```
{
  success: Boolean,
  refundedAmount: Number,
  errorCode: String (실패 시)
}
```

**사이드 이펙트**
- `예약내역`의 환불금액, 환불일시 업데이트
- 결제상태가 부분환불이면 PARTIAL_REFUNDED, 전액이면 REFUNDED
- Google Calendar 이벤트 삭제 또는 표시 변경
- 고객에게 환불 안내 발송
- `결제로그`에 REFUND_PROCESSED 기록

**제약사항**
- 환불 금액은 결제 금액을 초과할 수 없음
- 이미 전액 환불된 건은 추가 환불 불가
- 환불 정책에 따른 금액 자동 계산은 별도 함수 `calculateRefundAmount()`로 분리

### Task 0-4. 가용성 확인 함수 확장

**목적**  
현재 `checkAvailability`는 확정 예약만 고려. 임시 점유까지 포함해야 동시 결제 충돌 방지 가능

**요구 결과**

#### 4-1. `checkAvailability` 동작 변경

**기존 동작**: 입금확인=Y 예약만 차단  
**변경 후 동작**: 다음 모두 차단
- `예약내역`의 결제상태가 PAID, PARTIAL_REFUNDED 행

### Task 0-6. 에러 응답 포맷 통일

**목적**  
클라이언트가 에러 처리를 일관되게 할 수 있도록 모든 함수의 실패 응답 포맷 통일

**요구 결과**

#### 6-1. 표준 응답 포맷

성공:
```
{ success: true, data: {...} }
```

실패:
```
{
  success: false,
  errorCode: String,    // 식별자 (대문자 + 언더스코어)
  errorMessage: String, // 사용자에게 표시할 메시지 (한글)
  details: Object       // (선택) 디버그용
}
```

#### 6-2. 표준 errorCode 목록 (사전 정의)

| Code | 의미 |
|---|---|
| INVALID_INPUT | 입력 유효성 실패 |
| MISSING_REQUIRED_FIELD | 필수 필드 누락 |
| RESERVATION_NOT_FOUND | 예약 미존재 |
| ORDER_NOT_FOUND | 주문 미존재 |
| ORDER_EXPIRED | 주문 만료 |
| TIME_BLOCKED | 시간대 점유됨 |
| AMOUNT_MISMATCH | 금액 불일치 |
| PG_API_ERROR | PG 호출 실패 |
| PG_DECLINED | PG 결제 거절 |
| INVALID_OPTION | 옵션 상품 오류 |
| STORAGE_ERROR | 시트 접근 실패 |
| PERMISSION_DENIED | 권한 부족 |
| RATE_LIMITED | 요청 과다 |
| UNKNOWN_ERROR | 기타 |

**적용 범위**  
기존 함수 중 클라이언트에서 호출 가능한 함수 모두 (`submitReservation`, `checkAvailability`, `getSettings` 등)는 위 포맷에 맞춰 응답하도록 정비. 단, 기존 호출자가 의존하는 필드(`reservationNumber` 등)는 `data` 내부로 옮기되 root에도 유지하여 하위 호환성 확보.

### Task 0-7. 로깅 표준화


## 4. 비기능 요구사항

### 4.1 보안

| 항목 | 요구사항 |
|---|---|
| 카드정보 | 절대 저장/전송하지 않음. PG사 결제창에서만 처리 |
| 시크릿 키 | Script Properties 저장. 코드/시트/로그 평문 노출 금지 |
| 금액 검증 | 클라이언트 전달 금액은 신뢰하지 않고 서버에서 재계산 |
| Webhook | 시그니처 검증 필수, 검증 실패 시 즉시 무시 |
| 권한 | 환불, 결제 취소 등 민감 작업은 관리자 인증 필요 |
| 개인정보 | 로그에 카드번호 패턴(`\d{4}-\d{4}-\d{4}-\d{4}`) 자동 마스킹 |

### 4.2 성능

| 항목 | 요구사항 |
|---|---|
| Apps Script 실행 시간 | 단일 함수 6분 이내 (스크립트 한계) |
| 사용자 응답 시간 | 가용성 조회 1초 이내, 결제 검증 5초 이내 |
| 동시 요청 | LockService로 중복 예약 방지 |
| Webhook 응답 | 즉시 200 응답 후 후처리는 비동기 (Trigger) |

### 4.3 확장성

| 항목 | 요구사항 |
|---|---|
| PG 교체 | 결제 모듈 인터페이스를 통해 토스→포트원 등 교체 시 코드 변경 최소화 |
| 옵션 상품 추가 | 시트에 행만 추가하면 자동 반영 (코드 변경 불필요) |
| 알림 채널 추가 | 슬랙, 디스코드 등 추가 시 알림 발송 함수만 확장 |
| 정기결제 (Phase 2) | 현재 구조에 빌링키 관리 시트 추가만으로 수용 가능해야 함 |

### 4.4 유지보수성

| 항목 | 요구사항 |
|---|---|
| 코드 주석 | 모든 공개 함수에 JSDoc 형식 주석 |
| 설정 외부화 | 정책 값(요금, 점유 시간, 환불율)은 설정 시트로 분리 |
| 테스트 가능성 | 결제 모듈은 mock PG 응답으로 단위 테스트 가능 |
| 문서 동기화 | 본 명세서 변경 시 코드 주석도 동시 업데이트 |
