# Phase 0 배포 가이드

> Apps Script 9개 .gs 파일 반영 + 스프레드시트 스키마 마이그레이션 절차.
> 본 계약 진입 시 결제 모듈을 신속하게 통합하기 위한 가계약 단계 선작업이다.

---

## 0. 사전 준비 — 백업 (5분)

작업을 시작하기 전에 반드시 백업한다. 중간 단계에서 문제가 생겨도 되돌릴 수 있어야 한다.

### 0-1. 스프레드시트 백업
1. 운영 중인 예약 관리 스프레드시트를 연다.
2. 메뉴: `파일 → 사본 만들기`
3. 사본 이름: `예약관리시스템_백업_260517_Phase0이전`
4. 본인 드라이브에 저장.

### 0-2. Apps Script 코드 백업
1. 운영 스프레드시트에서 `확장 프로그램 → Apps Script`로 진입.
2. 좌측 파일 목록에서 `Code.gs` 클릭.
3. 전체 내용을 복사하여 로컬에 텍스트 파일로 저장 (`Code.gs.backup_260517.txt` 등).

> 참고: 이번 작업이 끝나면 Apps Script의 `Code.gs` 파일은 삭제된다. 9개 파일로 분리되기 때문이다.

## 1. .gs 파일을 Apps Script에 반영 (15분)

### 1-1. Apps Script 에디터 열기
스프레드시트 → `확장 프로그램 → Apps Script` → 에디터 진입.

### 1-2. 신규 파일 9개 생성

좌측 파일 목록 상단의 `+` 버튼 → `스크립트` 클릭 → 파일명 입력. 다음 **순서대로** 9개 파일을 만든다 (Apps Script는 알파벳 순으로 로드하므로 숫자 prefix가 의미가 있다).

| # | 파일명 (Apps Script에 입력) |
|---|---|
| 1 | `00_Entry` |
| 2 | `10_Config` |
| 3 | `20_Util` |
| 4 | `30_Reservation` |
| 5 | `40_Pricing` |
| 6 | `50_Availability` |
| 7 | `60_Payment` |
| 8 | `70_Integration` |
| 9 | `90_Test` |

> Apps Script는 `.gs` 확장자를 자동으로 붙이므로 파일명에 `.gs`를 입력하지 않는다.

### 1-3. 각 파일에 내용 복사

이 저장소(`02_IhadaReservation` 폴더)의 동일 이름 파일을 열어 **내용 전체를 복사**한 후 Apps Script 에디터의 해당 파일에 **붙여넣기**한다. 각 파일을 붙여넣은 직후 `Ctrl+S` (또는 `Cmd+S`)로 저장.

> ⚠️ **이 시점에 함수를 실행하지 말 것**. 9개 파일이 모두 들어가기 전까지는 새 함수와 기존 `Code.gs`의 동일 이름 함수가 충돌 상태다.

### 1-4. 기존 Code.gs 삭제

9개 파일이 모두 반영된 직후, 같은 자리에서 즉시 진행한다.

1. 좌측 파일 목록에서 `Code` 파일에 마우스 호버 → `⋮` (점 3개) 메뉴.
2. `삭제` 클릭 → 확인.

> 이 단계가 끝나야 함수 이름 충돌이 사라지고 실행이 안전해진다.

### 1-5. 매니페스트 확인 (선택)

좌측 톱니바퀴 → `프로젝트 설정` → "appsscript.json 매니페스트 파일을 에디터에 표시" 체크. 새로 권한이 필요한 API는 없으므로 이미 동작하던 권한 그대로 충분하다.

## 4. 스프레드시트 스키마 마이그레이션 (3분)

명세서 Task 0-2: 신규 시트 3개 + 예약내역 컬럼 9개 + 설정 6개 항목 추가.

### 4-1. 사전 가드 확인

**V열(22번째)이 `입금확인`인지** 시트에서 직접 눈으로 확인한다. 다른 값이면 컬럼이 이미 시프트된 상태이므로 작업을 중단하고 백업 본으로 돌아가야 한다.

### 4-2. 실행
1. Apps Script 에디터 상단 함수 드롭다운에서 `initializePaymentSchema` 선택.
2. ▶ `실행`.
3. 실행 로그(`보기 → 실행`)에서 반환값 확인:
   ```json
   {
     "addedColumns": [{"col":28,"name":"옵션상품"}, ...],
     "createdSheets": ["옵션상품", "임시주문", "결제로그"],
     "addedSettings": ["임시점유시간", "결제재시도횟수", ...]
   }
   ```

### 4-3. 스프레드시트 결과 확인

스프레드시트 탭에서 직접 눈으로 검증:

**예약내역 시트**
- 컬럼이 AA(27번)에서 AJ(36번)까지 확장됨.
- 헤더: ... `Z:알림톡발송상태` `AA:비고` `AB:옵션상품` `AC:옵션금액` `AD:주문번호` `AE:결제ID` `AF:결제수단` `AG:결제상태` `AH:결제완료일시` `AI:환불금액` `AJ:환불일시`.
- 기존 행의 A~AA 데이터는 그대로 보존되어 있어야 함.
- AB~AJ 열은 기존 행에 대해 빈 셀.

**신규 시트 (탭 하단에 3개 추가됨)**
- `옵션상품`: 헤더 9개 — 상품ID/카테고리/상품명/설명/가격/단위/최대수량/활성화/표시순서
- `임시주문`: 헤더 10개 — 주문번호/생성일시/만료일시/예약날짜/시작시간/종료시간/Room타입/예약데이터/금액/상태
- `결제로그`: 헤더 9개 — 로그ID/일시/주문번호/이벤트유형/결제ID/금액/응답코드/응답메시지/상세

**설정 시트**
- 기존 항목 그대로.
- 끝에 6개 행 추가:
  - `임시점유시간 / 10 / 분`
  - `결제재시도횟수 / 3 / 회`
  - `결제재시도간격 / 5 / 초`
  - `환불정책_24시간이상 / 100 / %`
  - `환불정책_12시간이상 / 50 / %`
  - `환불정책_12시간미만 / 0 / %`

### 4-4. 재실행 안전성 (idempotent)
실수로 `initializePaymentSchema`를 한 번 더 실행해도 데이터가 망가지지 않는다. 헤더/시트/설정 키가 이미 존재하면 건너뛴다. `added*` 배열이 빈 채로 반환된다.

## 5. 자동 검증 — `_verifyPhase0()` (1분)

명세서 5절 인수 기준을 코드로 일괄 점검.

1. Apps Script 에디터 상단 함수 드롭다운에서 `_verifyPhase0` 선택.
2. ▶ `실행`.
3. 실행 로그 확인. 다음 형태로 출력됨:
   ```
   === _verifyPhase0 결과 ===
   통과: 47 / 47
   ```
4. **단 1건이라도 실패가 있으면** 실패 항목 목록이 함께 출력된다. 가장 흔한 실패:
   - `cols.V=입금확인` 실패 → V열이 시프트된 환경. 작업 중단.
   - `sheet:옵션상품` 등 실패 → `initializePaymentSchema`를 실행하지 않았거나 권한 부족.
   - `price.basePrice` 등 실패 → 설정 시트의 `기준인원/시간당기본요금/추가인원단가/VAT요율` 값이 비어있음.

### 마스킹 검증 (선택)
1. 함수 드롭다운에서 `_testMasking` 선택 → 실행.
2. 스프레드시트 `결제로그` 시트의 마지막 행을 열어 I열(상세) 값 확인.
3. 카드번호 `1234-5678-9012-3456`이 `1234-****-****-3456` 형태로 마스킹되어 있어야 함. 시크릿 키 값이 `***`로 가려져 있어야 함.

### 6-2. 회귀 테스트 시나리오

| # | 시나리오 | 기대 결과 |
|---|---|---|
| 1 | 폼 로드 | F12 콘솔에 `설정 로드 완료` / `Room 정보 로드 완료` 출력. 폼이 정상 렌더링 |

## 8. 운영 환경 적용 후속 작업 (관리자 작업, 수시)

Phase 0은 자리만 만든다. 실제 운영 가치를 내려면 다음 키를 채워야 한다.

### 8-1. 즉시 채워야 하는 키 (Phase 0)
- `ADMIN_EMAIL` — 관리자 이메일 (Phase 1에서 알림 수신용)
- `ADMIN_PHONE` — 010-0000-0000 형식
- `CALENDAR_ID_CONFIRMED` — 확정 예약용 Calendar ID. 빈 값이면 기본 Calendar 사용. 별도 Calendar로 분리하고 싶으면 입력.
- `TERMS_DOC_ID` — `15NN03jeN6CHls3TMjkTIQWfmQIDvnxuoKukARue3f_0`. Property에 명시 등록해두면 추후 약관 문서 교체 시 코드 수정 없이 ID만 바꾸면 됨.

### 8-2. Phase 1 시점에 채울 키 (지금 비워둠)
- `PAYMENT_CLIENT_KEY` — 토스페이먼츠 발급
- `PAYMENT_SECRET_KEY` — 토스페이먼츠 발급
- `PAYMENT_WEBHOOK_SECRET` — 토스페이먼츠 Webhook 시그니처용
- `KAKAO_API_KEY` / `KAKAO_SENDER_KEY` — 알림톡 발송용

### 8-3. Property 입력 방법
1. Apps Script 에디터 → 좌측 톱니바퀴 `프로젝트 설정`.
2. `스크립트 속성` 섹션에서 키 옆 연필 아이콘 클릭.
3. 값 입력 → 저장.

> ⚠️ Property 값을 입력한 직후, 본 가이드 5절 자동 검증을 한 번 더 돌려 키가 캐싱 문제 없이 읽히는지 확인 권장.

## 9. 롤백 방법

문제가 생겨 Phase 0 이전 상태로 돌리고 싶을 때:

### 9-1. 코드 롤백
1. Apps Script 에디터에서 9개 신규 파일을 모두 삭제.
2. 좌측 `+` → `스크립트` → 파일명 `Code` 입력.
3. 0-2에서 백업한 `Code.gs.backup_260517.txt` 내용을 붙여넣기 → 저장.
4. `배포 → 배포 관리` → 새 버전 → 배포.

### 9-2. 스프레드시트 롤백
**예약내역 시트의 AB~AJ 컬럼은 그대로 두어도 무방**하다. 기존 코드는 이 컬럼들을 읽지도 쓰지도 않는다. 단, 시각적으로 거슬리면 컬럼을 우클릭 → 삭제.

신규 시트 3개(`옵션상품`, `임시주문`, `결제로그`)도 그대로 두면 됨. 비어있으므로 동작에 영향 없음.

설정 시트의 새 키 6개도 그대로 두면 됨.

> 즉, 데이터는 모두 idempotent하게 추가되어 있으므로 코드만 롤백하면 충분하다. 강제 삭제는 권장하지 않는다.

### 9-3. 최악의 경우 — 백업 본으로 통째로 교체
0-1에서 만든 스프레드시트 사본을 운영용으로 승격. 단, 그 사이 신규 예약이 있었다면 수동으로 옮겨와야 함.

## 10. 트러블슈팅

| 증상 | 원인 | 해결 |
|---|---|---|
| 함수 실행 시 `ReferenceError: getSheet is not defined` | 파일 9개 중 일부 누락 또는 `20_Util.gs` 내용 누락 | 좌측 파일 목록에서 9개 모두 존재하는지 확인. `20_Util` 내용 다시 붙여넣기 |
| `initializePaymentSchema` 실행 시 `스키마 가정 깨짐: V열(22번째)이 입금확인이 아닙니다` | 예약내역 시트 컬럼이 이전에 변형됨 | 백업 본의 헤더와 비교. 안전을 위해 백업 본 복원 후 재시도 |
| 클라이언트가 `result.basePrice`를 못 읽음 (가격이 0원으로 표시) | 응답이 `data:{basePrice:..}`로 들어가고 root에 노출 안 됨 | `40_Pricing.gs`의 `calculatePrice` 함수의 `ok(view, view)` 호출 부분 확인. 두 번째 인자(rootShim)가 누락되면 이 증상 |
| 약관이 안 뜸 | `TERMS_DOC_ID` Property 값이 잘못된 ID로 입력됨 | Property 값을 비우면 자동으로 하드코딩 폴백 사용. 또는 Docs URL에서 정확한 ID 재추출 |
| 예약현황로그 시트가 자꾸 생김 | `checkAvailability`가 호출될 때마다 시트가 없으면 자동 생성 (기존 동작 유지) | 정상 동작. 무시 |
| `_verifyPhase0` 결과 중 일부 실패하지만 회귀 시나리오는 정상 | 설정 시트의 값이 명세서 기대값과 다른 경우 정상 | 7번째 체크(`price.basePrice`)가 실패해도 회귀 시나리오 3에서 정상 가격이 나오면 운영상 문제 없음. 검증 함수의 `expected` 값을 운영 환경에 맞게 조정 가능 |
